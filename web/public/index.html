How to Calibrate the Mount

Calibration aligns the mount's internal model with the real sky. You need at least 3 reference objects (stars) with known coordinates. The more measurements you add, the better the accuracy.

1 Add reference measurements (Bootstrap)
  1. In the Initial Calibration (Bootstrap) card below, click "Add Reference Measurement" to expand the search section.
  2. Type a star name or catalog ID (e.g. Polaris, Sirius, Vega) and click "Search".
  3. Click on an object from the search results to select it — its coordinates auto-fill into the form.
  4. Click "Slew & Add Measurement" to slew the mount to that object and record the measurement, or "Add Measurement Direct" if the mount is already pointing at it.
  5. Repeat for at least 3 objects spread across the sky.
2 Run the initial calibration

Once you have enough measurements, click the "Run Initial Calibration" button in the Bootstrap card. The controller computes the alignment transform. The "TPOINT Ready" field will show Yes when the mount is ready for precise calibration.

You can track the Alignment Error and RMS Residual — lower values mean better alignment.

3 Add precise measurements (TPOINT)
  1. In the Precise Calibration (TPOINT) card, use the same search workflow to add more reference measurements.
  2. Add measurements for 10–20 objects distributed evenly across the sky for the best pointing model.
  3. Each measurement records the difference between the expected and actual position of the object.
4 Run TPOINT calibration

Click "Run TPOINT Calibration" to compute the full pointing model. The TPOINT model fits up to 6 error terms (cone, hub, collimation, etc.) and reports:

  • Residual RMS — typical pointing error after correction
  • Max Residual — worst-case error
  • Chi-Squared — overall model fit quality (lower is better)
5 Tips
  • Spread objects across the sky — avoid clustering them in one area.
  • Use the "Clear Measurements" button to reset and start over if needed.
  • Click "Refresh Status" to manually update the displayed statistics.
  • After calibration, use the Control tab's calibrated nudge buttons for fine adjustments.
  • Re-run calibration if you change the setup (new location, different load, etc.).

📋 Supported File Formats

✅ CSV (Comma-Separated Values) — w pełni obsługiwany

Jedyny format w pełni zaimplementowany po stronie serwera. System korzysta z mapowania pól (field_mapping), które dopasowuje kolumny CSV do pól obiektów astronomicznych. Separator wykrywany jest automatycznie (; lub ,).

📌 Kody typów: Obsługuje standardowe kody OpenNGC — Gx (galaktyka), OC (gromada otwarta), Gb (gromada kulista), Pn (mgławica planetarna), Nb (mgławica dyfuzyjna), * (gwiazda), SNR (pozostałość po supernowej), QSO (kwazar) i wiele innych.
📌 Konstelacja: Używaj standardowych 3-literowych kodów IAU (np. And, Ori, UMa). Konstelacja jest widoczna zarówno na liście obiektów, jak i w szczegółowym widoku.

⚠️ JSON — niezaimplementowany

Format zdefiniowany w protokole gRPC, ale nie jest jeszcze obsługiwany przez serwer. Docelowo spodziewana struktura: 

{
      "objects": [
        {
          "name": "M31",
          "ra_hours": 0.7123,
          "dec_degrees": 41.269,
          "v_magnitude": 3.44,
          "constellation": "And",
          ...
        }
      ]
    }

Każdy obiekt odpowiada polom zdefiniowanym w AstronomicalObject protobuf.

⚠️ FITS (Flexible Image Transport System) — niezaimplementowany

Standardowy format używany w astronomii do przechowywania danych obrazowych i tabelarycznych. Docelowo system będzie odczytywać tabele binarne (BINTABLE) z rozszerzeń FITS zawierające kolumny z katalogami obiektów (np. katalogi z misji Gaia, 2MASS, SDSS).

Wymagana będzie osobna biblioteka (cfitsio) do parsowania formatu.

⚠️ VOTable (XML) — niezaimplementowany

Format XML przyjęty przez Międzynarodową Unię Astronomiczną (IAU) do wymiany danych tabelarycznych. Docelowo system będzie parsować strukturę <RESOURCE> → <TABLE> → <FIELD> / <DATA> → <TR> / <TD> z mapowaniem typów VOTable na pola protobuf.

Wykorzystywany przez serwisy takie jak VizieR, SIMBAD i IRSA.

⚠️ SIMBAD — niezaimplementowany

Format eksportu z bazy danych SIMBAD (CDS, Strasbourg). Docelowo system będzie akceptować wyniki zapytań w formacie tekstowym SIMBAD, zawierające:

  • główny identyfikator obiektu,
  • typ obiektu według klasyfikacji SIMBAD,
  • współrzędne równikowe (J2000),
  • wielkości gwiazdowe (V, B),
  • klasę widmową,
  • oznaczenia w innych katalogach.

⚠️ NED (NASA/IPAC Extragalactic Database) — niezaimplementowany

Format eksportu z bazy danych NASA/IPAC zawierającej obiekty pozagalaktyczne. Docelowo system będzie akceptować wyniki zapytań NED w formacie tekstowym, zawierające nazwy obiektów, współrzędne, przesunięcie ku czerwieni (redshift), wielkości gwiazdowe, morfologię galaktyk i odległości.

⚠️ MPC (Minor Planet Center) — niezaimplementowany

Standardowy format używany przez Minor Planet Center do publikacji danych orbitalnych planetoid i komet. Docelowo system będzie parsować format MPC 80-column (lub rozszerzony MPC Packed), zawierający:

  • tymczasowe i stałe oznaczenie obiektu,
  • elementy orbitalne (a, e, i, Ω, ω, M, epoka),
  • bezwzględną wielkość gwiazdową (H),
  • parametr nachylenia jasności (G),
  • datę ostatniej obserwacji.
